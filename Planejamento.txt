========================================
MÓDULO 1 — SERVIÇO DE E-MAIL
========================================

Por que primeiro: todos os outros módulos dependem de envio de e-mail.

Serviço: Microsoft Exchange da empresa via Nodemailer com SMTP autenticado.

Exchange é o servidor de e-mail da Microsoft usado em ambientes corporativos.
Pode ser de dois tipos — verificar qual a empresa usa:

  TIPO 1 — Exchange Online (Microsoft 365 / Office 365, nuvem):
    SMTP_HOST = smtp.office365.com
    SMTP_PORT = 587
    Como identificar: o Outlook Web App abre em *.office365.com ou *.outlook.com

  TIPO 2 — Exchange Server on-premises (servidor local da empresa):
    SMTP_HOST = endereço do servidor interno (perguntar ao TI)
                exemplo: mail.filtrovali.com.br ou IP do servidor
    SMTP_PORT = 587 (ou 25 se não exigir autenticação na rede interna)
    Como identificar: o Outlook Web App abre em um domínio próprio da empresa

  Comum a ambos:
    SMTP_SECURE = false  (usa STARTTLS na porta 587)
    SMTP_USER   = e-mail da conta remetente
    SMTP_PASS   = senha da conta (ou App Password se MFA estiver ativo)
    SMTP_FROM   = nome exibido + e-mail (ex: "Filtrovali <email@empresa.com.br>")
    APP_URL     = URL base do sistema (muda quando entrar em produção)

ATENÇÃO — verificar antes de implementar:
  - Exchange Online: o tenant M365 precisa ter "SMTP AUTH" habilitado para a conta.
    Centro de administração M365 → Usuários → [conta] → E-mail →
    Configurações de e-mail → SMTP Autenticado: ATIVAR.
  - Se a conta tiver MFA, gerar uma "App Password" nas configurações de segurança.
  - Exchange on-premises: verificar se o servidor permite relay SMTP para
    o IP da máquina onde o backend vai rodar.
  - Rodar o script de diagnóstico antes de desenvolver os outros módulos.

Script de teste já criado: backend/scripts/test-email.js
  Instalar nodemailer e executar:
    npm install nodemailer
    node scripts/test-email.js
  O script tenta conectar, envia um e-mail de teste e exibe diagnóstico
  detalhado em caso de falha (auth negada, relay bloqueado, etc.)

Arquivos a criar:
  - backend/src/lib/mailer.js          → instância do transporter Nodemailer
  - backend/src/lib/email-templates.js → funções que retornam HTML de cada template

MÓDULO 3 — RECUPERAÇÃO DE SENHA (TODAS AS CONTAS)
========================================

A função "Esqueci minha senha" deve funcionar para TODOS os tipos de conta:
internos (COLLABORATOR, MANAGER) e clientes (CLIENT).

Além da recuperação de senha, o sistema deve oferecer a função
"Trocar senha" para TODOS os tipos de conta autenticada:
  - COLLABORATOR
  - MANAGER
  - CLIENT

SCHEMA (Prisma)
  Adição no model User:
    email  String?

  Novo model:
    PasswordResetToken {
      id        String    @id @default(cuid())
      tokenHash String    @unique
      userId    String
      expiresAt DateTime              (1 hora de validade)
      usedAt    DateTime?
      createdAt DateTime  @default(now())
      user      User      @relation(...)
    }

FLUXO PARA USUÁRIOS INTERNOS (COLLABORATOR / MANAGER)
  - Login permanece por username, com opção de login também por e-mail
  - Para recuperação, usuário pode informar username ou e-mail
  - Sistema localiza o usuário por:
      - username, ou
      - User.email
  - Gera token seguro (crypto.randomBytes), salva hash no banco
  - Envia e-mail para User.email com link de recuperação
  - Link: APP_URL/reset-password?token=xxx
  - Página de reset: formulário de nova senha + confirmação
  - Ao confirmar: valida token (não expirado, não usado), atualiza passwordHash,
    marca usedAt

FLUXO PARA CLIENTES (CLIENT)
  - A conta CLIENT é vinculada ao CNPJ limpo
  - O e-mail de recuperação é sempre o clientEmailPrimary
  - Usuário digita o CNPJ (username do cliente)
  - Sistema localiza todos os projetos com clientCnpj = username
  - Obtém o e-mail principal do cliente a partir dos projetos vinculados
  - Envia o link de recuperação somente para o e-mail principal
  - O restante do fluxo é idêntico ao de internos
  - Os e-mails em cópia não participam da recuperação de senha

FRONTEND
  - Tela de login: link "Esqueci minha senha"
  - Tela de solicitação: campo para username ou e-mail (para internos)
    ou campo para CNPJ (detectado automaticamente pelo formato)
  - Página /reset-password?token=xxx: formulário nova senha + confirmação
  - Área autenticada: opção "Trocar senha" para todos os usuários
    com fluxo de:
      - senha atual
      - nova senha
      - confirmação da nova senha

NOTA: A URL base do link muda com a variável APP_URL quando entrar em produção.

BACKEND (novos endpoints em /auth)
  - POST /auth/forgot-password  → gera token e envia e-mail
  - POST /auth/reset-password   → valida token e atualiza senha
  - POST /auth/change-password  → exige autenticação, valida senha atual
    e atualiza a senha do usuário logado

MÓDULO 4 — CONTAS DE CLIENTE (CRIAÇÃO AUTOMÁTICA)
========================================

SCHEMA (Prisma)
  UserRole enum: adicionar valor CLIENT

  O campo email do User (adicionado no Módulo 3) não é usado diretamente
  para login do cliente — a recuperação vai para o e-mail principal do projeto.

LÓGICA DE IDENTIFICAÇÃO
  O username do cliente é o CNPJ limpo (só dígitos, 14 chars).
  Ao buscar projetos do cliente, o sistema filtra Project onde
  clientCnpj (sem formatação) = username do User logado.

CRIAÇÃO AUTOMÁTICA
  Trigger: toda vez que um projeto é criado ou atualizado E clientEmailPrimary
  está preenchido:

  1. Limpar clientCnpj do projeto (remover pontos, barras e traço → 14 dígitos)
  2. Verificar se já existe User com username = cnpjLimpo e role = CLIENT

  CASO: NÃO existe ainda
    - Criar User com username = cnpjLimpo, role = CLIENT
    - Gerar senha inicial temporária de 6 dígitos
    - Enviar e-mail somente para clientEmailPrimary com:
        - usuário (CNPJ formatado)
        - senha gerada
        - orientação de que a troca de senha estará disponível dentro do sistema
    - clientEmailCc recebe apenas notificação opcional de criação de acesso,
      sem função de gestão da conta

  CASO: JÁ existe
    - Não recriar nem resetar senha
    - Se for um projeto NOVO sendo vinculado a um CNPJ já cadastrado:
      enviar e-mail para clientEmailPrimary informando que um novo contrato/projeto
      foi associado à conta do cliente
    - clientEmailCc pode receber cópia dessa notificação, se desejado

LOGIN COM FORMATAÇÃO DE CNPJ
  - No campo de username, se o valor digitado tiver 14 dígitos numéricos,
    aplicar máscara visual: XX.XXX.XXX/XXXX-XX
  - O valor submetido à API é sempre o CNPJ sem formatação (14 dígitos)
  - Após login como CLIENT, redirecionar para o Painel do Cliente

ABA DE USUÁRIOS (frontend) — MÓDULO 4B
  A seção de gerenciamento de usuários será dividida em duas sub-abas:
    - Internos: lista atual de COLLABORATOR e MANAGER
    - Clientes: lista de usuários CLIENT com CNPJ formatado, projetos vinculados
      e ação de reenviar credenciais


========================================
MÓDULO 5 — PAINEL DO CLIENTE
========================================

ESTRUTURA DE ABAS
  Cada aba corresponde a um projeto vinculado ao CNPJ do cliente.
  Título da aba:
    [contractCode] — Missão [code] — [name]
  Onde:
    contractCode = número da proposta (campo contractCode do Project)
    code         = número da missão  (campo code do Project)
    name         = nome do projeto   (campo name do Project)

  Cada aba exibe de forma independente apenas os relatórios daquele contrato.

VISIBILIDADE DOS RELATÓRIOS
  +------------------------------------------+-------------------------------------------+
  | Situação                                 | O que o cliente vê                        |
  +------------------------------------------+-------------------------------------------+
  | RDO aprovado pelo gestor                 | RDO aparece na lista                      |
  | RDO pendente ou devolvido                | Invisível para o cliente                  |
  | RDO assinado pelo cliente                | Relatórios de serviço associados aparecem |
  +------------------------------------------+-------------------------------------------+

AÇÕES DO CLIENTE SOBRE UM RDO
  - Baixar PDF: endpoint de geração existente, liberado para role CLIENT
    sobre relatórios dos seus projetos
  - Comentar: campo de texto livre associado ao relatório
  - Aprovar ou Reprovar: registrado no banco com rastreabilidade
  - Ao aprovar, o RDO entra no estado final "ASSINADO"
  - Após "ASSINADO", o relatório fica bloqueado para qualquer edição:
      - colaborador não edita
      - gestor não altera
      - cliente não altera
  - Após "ASSINADO", o relatório apenas pode ser consultado/baixado
    e eventualmente arquivado por regra administrativa

SCHEMA — AÇÕES DO CLIENTE
  Novo model ClientReportReview:
    id            String              @id @default(cuid())
    reportId      String
    clientUserId  String
    action        ClientReviewAction  (enum: APPROVED | REJECTED)
    comment       String?
    ipAddress     String?
    userAgent     String?
    createdAt     DateTime            @default(now())
    report        Report              @relation(...)
    user          User                @relation(...)

  Ajuste recomendado no model Report:
    adicionar um estado final explícito para assinatura do cliente
    Exemplo:
      ReportStatus:
        PENDING
        APPROVED
        RETURNED
        SIGNED

  Regra de negócio:
    - APPROVED = aprovado pelo gestor
    - SIGNED   = aprovado/assinado pelo cliente
    - SIGNED é estado terminal do fluxo

VALIDADE JURÍDICA DA APROVAÇÃO — MVP (Opção A)
  Ao clicar em Aprovar, o cliente vê uma caixa de confirmação explícita com texto:
    "Eu, [nome do cliente/empresa], CNPJ [XX.XXX.XXX/XXXX-XX], declaro que li
     e aprovo o [tipo] nº [número] referente ao projeto [nome], em [data/hora]."
  O sistema registra no banco:
    - Ação (APPROVED/REJECTED)
    - Comentário (opcional)
    - IP do cliente
    - User Agent do navegador
    - Timestamp exato
  Esses dados formam uma trilha de auditoria com valor como evidência.

VALIDADE JURÍDICA — FUTURO (Opção C)
  Integração com plataforma de assinatura eletrônica com validade jurídica
  pela MP 2.200-2/2001. Plataforma a avaliar: ZapSign (zapsign.com.br).
  O fluxo seria: ao aprovar, gerar PDF do RDO e enviar via API do ZapSign,
  que retorna link para o cliente assinar digitalmente.
  Implementação planejada para após o MVP estar estável.

ALTO — Corrigir antes de ir para produção
------------------------------------------

[S1] CORS aberto sem restrição de origem
  Arquivo: backend/src/app.js linha 24
  Problema: sem definir "origin", qualquer domínio faz requisições à API.
  Correção: cors({ origin: process.env.ALLOWED_ORIGIN, exposedHeaders: [...] })
  Nova variável de ambiente: ALLOWED_ORIGIN=https://dominio.empresa.com.br

[S2] Fotos de relatórios servidas sem autenticação
  Arquivo: backend/src/app.js linha 52
  Problema: app.use('/relatorios', express.static(...)) é completamente público.
    Apenas /Assinaturas está protegida. Qualquer pessoa com o caminho acessa as fotos.
  Correção (curto prazo): estender a proteção de bearerToken para cobrir toda a
    rota /relatorios, não apenas /Assinaturas.

[S3] Sem rate limiting no endpoint de login
  Arquivo: backend/src/routes/resources/auth.js
  Problema: sem limitação de tentativas, ataques de força bruta são triviais.
    Combinado com S4 (senhas de 4 dígitos), é especialmente perigoso.
  Correção: instalar express-rate-limit. Aplicar no POST /api/auth/login.
    Sugestão: máximo de 10 tentativas por IP em janela de 15 minutos.

[S4] Senha automática de clientes muito fraca (4 dígitos numéricos)
  Problema: apenas 10.000 combinações possíveis.
  Correção: mínimo de 8 caracteres alfanuméricos aleatórios.
    Exemplo: Math.random().toString(36).slice(2, 10).toUpperCase()
    (gera ~2,8 trilhões de combinações)

[S5] Credenciais padrão no Docker Compose
  Arquivo: docker-compose.yml
  Problema: POSTGRES_PASSWORD: postgres exposta no repositório.
  Correção: POSTGRES_PASSWORD: ${POSTGRES_PASSWORD} e definir no .env do servidor.
    Nunca commitar credenciais reais no repositório.

[S6] Ausência de headers de segurança HTTP (Helmet.js)

[S7] Token de autenticação aceito via query string (?t=)
  Arquivo: backend/src/app.js linha 32

P2 — INTEGRAÇÃO ZAPSIGN (Assinatura Digital)
========================================

Plataforma: ZapSign (zapsign.com.br) — assinatura eletrônica com validade
jurídica plena pela MP 2.200-2/2001.

Modelo de cobrança: por documento assinado — verificar planos antes de implementar.

FLUXO PROPOSTO
--------------
  1. Cliente clica em "Aprovar e Assinar Digitalmente" no painel.
  2. Backend gera o PDF do RDO (código já existente).
  3. Backend faz POST para API ZapSign:
       - PDF como base64
       - Nome e e-mail do signatário (clientEmailPrimary)
       - URL do webhook de retorno
  4. ZapSign responde com doc_token + link de assinatura.
  5. Backend salva doc_token no banco.
  6. Frontend exibe o link em modal ou nova aba para o cliente assinar.
  7. Cliente assina na interface do ZapSign.
  8. ZapSign dispara webhook para backend com status "signed".
  9. Backend valida o payload, atualiza status do relatório para SIGNED.

SCHEMA — alterações no model Report
-------------------------------------
  zapsignDocToken  String?    (token do documento no ZapSign)
  zapsignSignedAt  DateTime?  (timestamp da assinatura confirmada)
  zapsignDocUrl    String?    (URL do documento final assinado)

NOVOS COMPONENTES BACKEND
--------------------------
  Arquivo: backend/src/lib/zapsign.js
    - sendToZapSign(pdfBuffer, signerName, signerEmail)
    - verifyWebhookSignature(payload, signature)

  Endpoint: POST /api/webhooks/zapsign
    - Não usa requireAuth (chamado pelo ZapSign)
    - Valida assinatura HMAC do payload
    - Atualiza status do relatório para SIGNED

  Endpoint: POST /api/reports/:id/request-signature
    - Inicia o fluxo ZapSign para um RDO com status APPROVED
    - Retorna o link de assinatura

  Novas variáveis de ambiente:
    ZAPSIGN_API_TOKEN=...      (token da API ZapSign)
    ZAPSIGN_WEBHOOK_SECRET=... (valida autenticidade dos webhooks)

PRÉ-REQUISITOS ANTES DE IMPLEMENTAR
-------------------------------------
  [ ] Criar conta em zapsign.com.br e verificar planos/preços
  [ ] Gerar API Token no painel ZapSign
  [ ] Definir URL do webhook (backend precisa estar em produção com URL pública)
  [ ] Testar em sandbox antes de ativar em produção

P3 — DEPLOY EM PRODUÇÃO
========================================

AMBIENTE DEFINIDO
------------------
  Servidor físico Windows Server na empresa.
  Acesso externo habilitado (app visível na internet).
  Firewall físico já presente para proteção externa.
  Abordagem: Docker com containers Linux (via WSL2).

  O servidor também armazena arquivos internos da empresa —
  isolamento do Docker em relação ao host é ponto crítico.
  Ver seção "Isolamento e Segurança no Windows Server" abaixo.

ARQUITETURA
-----------
  Internet
      ↓
  Firewall físico (só portas 80 e 443 abertas para o servidor)
      ↓
  Windows Server
      ↓
  Docker (rede interna isolada: filtrovali-net)
    ├── nginx   → porta 80/443 do host  (único container exposto)
    ├── backend → porta interna 4000    (não exposto ao host)
    └── postgres→ porta interna 5432    (não exposto ao host)
      ↓
  Volumes Docker nomeados (isolados do restante do servidor)
    ├── filtrovali_pgdata      (dados do banco)
    ├── filtrovali_relatorios  (fotos e arquivos dos relatórios)
    └── filtrovali_assets      (logos e assets estáticos)

  O frontend não é servidor separado — o HTML único é servido
  pelo próprio Express (já configurado em app.js).

ISOLAMENTO E SEGURANÇA NO WINDOWS SERVER
-----------------------------------------
  Este é o ponto mais importante dado que o servidor tem arquivos internos.

  1. Volumes Docker: apenas as pastas de dados do APP são montadas.
     Os containers NÃO têm acesso ao restante do sistema de arquivos.
     Nunca montar C:\ ou diretórios de arquivos internos como volume.

  2. Rede Docker isolada: criar rede dedicada "filtrovali-net".
     backend e postgres não ficam expostos ao host nem à internet —
     apenas o nginx tem porta mapeada para o Windows (80 e 443).

  3. Portas expostas no Windows: SOMENTE 80 e 443.
     No Firewall do Windows Server: bloquear todas as outras portas
     de entrada vindas de IPs externos.
     A porta 5432 (PostgreSQL) e 4000 (backend) NUNCA devem ser
     abertas para fora — ficam restritas à rede interna do Docker.

  4. Usuário Docker: rodar o serviço Docker com uma conta Windows
     de baixo privilégio (não Administrator), sem acesso às pastas
     de arquivos internos da empresa.

  5. Firewall físico: configurar regras para que apenas as portas
     80 e 443 sejam encaminhadas para o IP do Windows Server.
     Todo o restante deve ser bloqueado antes de chegar ao servidor.

  6. Windows Defender + Windows Firewall: manter ativos como
     segunda camada, mesmo com o firewall físico.

  7. Atualizações: manter o Windows Server e o Docker atualizados —
     vulnerabilidades no Docker runtime são corrigidas regularmente.

  8. Monitoramento: configurar alertas de CPU/memória/disco e logs
     do nginx para detectar comportamento anormal (tentativas de
     acesso, volume de requisições, etc.).

PRÉ-REQUISITOS NO WINDOWS SERVER
----------------------------------
  [✓] Windows Server 2019 — confirmado. Suporta WSL2.
  [✓] Domínio registrado — confirmado.

  ATENÇÃO — Docker no Windows Server 2019:
    Docker Desktop NÃO deve ser usado em produção no Windows Server.
    É uma ferramenta para estações de trabalho, não para servidores.
    A abordagem correta é instalar o Docker Engine dentro do WSL2.

  Passos para habilitar Docker no Windows Server 2019:
    1. Habilitar WSL2 no Windows Server:
         dism.exe /online /enable-feature /featurename:Microsoft-Windows-Subsystem-Linux /all /norestart
         dism.exe /online /enable-feature /featurename:VirtualMachinePlatform /all /norestart
         (reiniciar o servidor após os dois comandos)
    2. Baixar e instalar o kernel do WSL2:
         https://aka.ms/wsl2kernel
    3. Definir WSL2 como padrão:
         wsl --set-default-version 2
    4. Instalar Ubuntu 22.04 via WSL2:
         wsl --install -d Ubuntu-22.04
    5. Dentro do Ubuntu WSL2, instalar Docker Engine:
         curl -fsSL https://get.docker.com | sh
         sudo usermod -aG docker $USER
    6. Instalar Docker Compose plugin:
         sudo apt install docker-compose-plugin
    7. Configurar Docker para iniciar com o WSL2:
         sudo systemctl enable docker

  A partir daí, todos os comandos docker e docker compose
  são executados dentro do terminal Ubuntu (WSL2).
  O Windows Task Scheduler pode chamar scripts WSL2 para
  automações (backup, reinicialização, etc.).

  [ ] IP fixo no servidor (ou DDNS — o UCG-Ultra tem DDNS nativo)
  [ ] Portas 80 e 443 encaminhadas no UCG-Ultra para o IP interno
      do Windows Server (UniFi Network → Port Forwarding)
      NUNCA encaminhar portas 4000 ou 5432 externamente.

  GATEWAY: UniFi Cloud Gateway Ultra (UCG-Ultra-EU)
    Não é empecilho. Funcionalidades úteis para o deploy:
    - Port Forwarding via interface gráfica do UniFi Network
    - Firewall integrado para bloquear portas não autorizadas
    - DDNS nativo (útil se o IP público da empresa for dinâmico)
    - IDS/IPS como camada extra de proteção
    Passos no UniFi:
      1. UniFi Network → Settings → Routing → Port Forwarding
      2. Criar regra: porta WAN 80  → IP do Windows Server, porta 80
      3. Criar regra: porta WAN 443 → IP do Windows Server, porta 443
      4. Firewall → criar regra bloqueando qualquer outra porta de entrada
         destinada ao IP do Windows Server vinda da WAN

========================================

SITUAÇÃO ATUAL
---------------
  Imagens salvas em:
    - Relatórios/Assinaturas/     → assinaturas dos colaboradores
    - Relatórios/Missão X/...     → fotos dos registros fotográficos

  Caminhos salvos como URLs relativas dentro do campo specialConditions
  (JSON) de cada relatório. PDF/DOCX lê os arquivos diretamente do disco.

OPÇÃO A — Manter como arquivos (recomendado para produção imediata)
--------------------------------------------------------------------
  Prós: código já funciona, performance excelente, sem alteração.
  Cons: backup precisa cobrir dois sistemas (banco + arquivos).

  Melhorias sem mudar a abordagem:
    - Em Docker: montar a pasta de relatórios como volume nomeado externo,
      garantindo persistência independente do container.
    - Mover a pasta para fora do diretório da aplicação:
        /var/filtrovali/relatorios  ao invés de  backend/Relatórios/
    - Incluir o volume no cron de backup diário.

OPÇÃO B — Armazenar no PostgreSQL como bytea
----------------------------------------------
  Prós: tudo em um lugar; backup do banco = backup completo.
  Cons: PostgreSQL não é otimizado para binários grandes. Banco cresce
    rápido (20 fotos por missão = ~10-40MB só em imagens). Geração de
    PDF precisaria de adaptação. Consultas ficam mais pesadas.

  Avaliação: NÃO recomendado para fotos de relatório.
  Poderia ser considerado apenas para assinaturas (~5-50KB cada).

OPÇÃO C — MinIO (armazenamento de objetos self-hosted, estilo S3)
------------------------------------------------------------------
  O que é: servidor compatível com AWS S3, roda como container Docker
  no mesmo servidor, tem interface web de gerenciamento.

  Prós: interface web, API S3-compatible, backup simples via sync de bucket,
    escalável, fácil migrar para AWS S3 no futuro se necessário.

  Cons: mais um serviço para gerenciar; geração de PDF/DOCX precisaria
    baixar imagens do MinIO antes de inserir no documento (leve latência extra);
    adaptação em toda a lógica de upload e leitura de arquivos.

RECOMENDAÇÃO
-------------
  Curto prazo (ir para produção agora):
    → Opção A com volume Docker externo.
    Sem mudança de código. Configure o volume no docker-compose.yml.
    Inclua no backup diário.

  Médio prazo (após estabilizar em produção):
    → Migrar para Opção C (MinIO).
    Adicionar MinIO no docker-compose.yml, refatorar upload e leitura
    para SDK do MinIO, migrar arquivos existentes para o bucket.

  Assinaturas especificamente:
    O campo Collaborator.signatureImage (String no schema) pode ser
    migrado para armazenar base64 diretamente no banco — pequena mudança
    pontual que elimina os arquivos de assinatura do disco sem afetar as fotos.

IMPACTO NA GERAÇÃO DE RELATÓRIOS
----------------------------------
  Opção A: nenhuma mudança no código de geração.

  Opções B ou C: as funções de leitura de imagens do disco em
    report-docx.js, report-rtp.js, report-rlq.js, report-rcp.js e
    report-rlm.js precisariam ser adaptadas para buscar dados binários
    da nova fonte antes de inserir no DOCX.
    Impacto estimado: moderado, afeta todos os tipos com fotos.
